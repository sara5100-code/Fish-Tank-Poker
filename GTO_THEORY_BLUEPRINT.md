# Fish Tank Poker GTO理論設計書

## 目的

Fish Tank Poker のレビュー精度を上げるため、まず「理論上のGTO基準」を独立したレイヤーとして定義する。

この設計では、最終評価をいきなりライブ実戦向けに作らない。先にGTO基準を作り、その上でライブ$2/$5、国内アミューズメントトーナメント、初心者矯正用の補正を重ねる。

理想のレビューは次の形にする。

```text
GTO基準では、この場面はベット45% / チェック55%の混合です。
ただしライブ$2/$5では、相手がリバーでブラフ不足になりやすいため、実戦推奨はチェック寄りです。
今回の判断はGTO上は許容、ライブ実戦ではかなり良い判断です。
```

## 基本原則

1. GTO基準とライブ補正を混ぜない
2. 評価は「実ハンド」ではなく「相手レンジ」に対して行う
3. 各アクション後にレンジを更新する
4. 頻度を出せる場面では、単一正解ではなく混合比率を表示する
5. ユーザー向け説明では、内部スコアではなく「なぜそのラインなのか」を口語で説明する

## 評価レイヤー

### 1. GTO Baseline Layer

理論上の基準。ここではライブ傾向や初心者矯正は入れない。

扱うもの:

- ポジション別プリフロップレンジ
- オープン、コール、3BET、4BET、5BET、オールインの基準頻度
- ポストフロップのレンジ優位
- ナッツ優位
- ベットサイズ候補
- 各サイズに対するベット頻度
- Call / Fold / Raise の混合頻度
- ブロッカーとコンボ数

出力例:

```js
{
  street: "flop",
  gtoMix: {
    check: 55,
    bet33: 30,
    bet75: 15
  },
  baselineAction: "mixed",
  confidence: "medium",
  reason: [
    "IPだがプリフロップ受け側",
    "トップペア中キッカー",
    "2トーンでプロテクション価値あり",
    "マルチウェイなので大サイズ頻度は落ちる"
  ]
}
```

### 2. Range State Layer

各プレイヤーがどのレンジを持っているかを追跡する。

プリフロップ開始時:

- UTG
- MP
- CO
- BTN
- SB
- BB

各アクション後:

- fold したレンジを除外
- call したレンジを残す
- raise したレンジを強いレンジと一部ブラフに分解
- check したレンジから強すぎる手や弱すぎる手を一部減らす
- bet したレンジをバリュー、ブラフ、プロテクションに分ける

重要:

```text
相手の実ハンドが見えていても、このレイヤーには入れない。
リザルト確認用の実ハンドと、評価用レンジは完全に分離する。
```

### 3. Board Texture Layer

ボードをGTO評価用に分類する。

分類軸:

- ペアボード
- トリップスボード
- モノトーン
- 2トーン
- 4枚同色
- 連結度
- A/K/Qハイ
- ローカードボード
- ストレート完成候補
- フラッシュ完成候補
- ナッツがどちらのレンジに多いか

例:

```text
Th 2c 5h
完全な静的ボードではない。
現状トップペアは強いが、ハート、A/K/Q/J/9/8/7/6/4/3で価値が変わりやすい。
GTO上はチェックも混ざるが、IPで全員チェック後なら中サイズベットも自然。
```

### 4. EV Approximation Layer

本物のソルバーを毎回回せないため、近似EVを作る。

EVを分解する:

- Fold EV
- Showdown EV
- Realization EV
- Future Street Risk
- Reverse Implied Odds
- Stack Commitment Risk

注意:

```text
Raw Equity はGTO評価の一部でしかない。
Raw EQ 70%でも、OOP・マルチウェイ・完成ボード・深いSPRなら大きく打てないことがある。
```

### 5. Strategy Mix Layer

単一推奨ではなく頻度を持つ。

出力形式:

```text
GTO頻度:
Check 55%
Bet 33% pot 30%
Bet 75% pot 15%
```

初心者向け表示:

```text
この場面はチェックとベットが両方あります。
標準はややチェック寄りですが、相手が下のペアやドローでコールしやすいなら33〜50%potで打てます。
```

## ライブ補正レイヤー

GTO基準を出した後でだけ使う。

### ライブ$2/$5補正

- リバー大サイズはバリュー過多
- チェックレイズブラフは少なめ
- マルチウェイでブラフ成功率低下
- コール過多相手にはブラフ減、バリュー増
- パッシブ相手のリバーベットにはワンペアを絞る
- ルース相手には薄バリューを増やす

### 初心者矯正補正

- 弱いオフスーツブロードウェイの参加を厳しくする
- 非BBコールドコールを厳しくする
- ワンペアのリバーコールを厳しくする
- ナッツでない強手のオールイン過信を抑える
- ベットサイズを固定セットから選ばせる

### トーナメント補正

- ICM
- バブル圧
- カバー関係
- BBアンティ
- 12〜25BBのreshove
- FTのペイジャンプ
- HUの広いレンジ

## レビュー文の理想形

### 悪い例

```text
複数のラインが成立する場面です。チェックは許容範囲ですが、推奨ベットを意識してください。
```

### 良い例

```text
GTO基準では、ここはチェックと小さめベットが混ざります。
あなたはトップペアを持っていますが、相手のレンジにも強いTxやドローが残っています。
チェックでポットを抑える考え方と、33〜50%potで下のペアやドローから取る考え方の両方があります。
ライブ$2/$5なら、相手がチェック後に弱い手でコールしやすいので、小さめベット寄りに少しズラします。
```

## 実装順序

### Phase 1: GTO基準の出力枠を作る

- `gtoBaseline`
- `gtoMix`
- `gtoReason`
- `solverConfidence`
- `liveAdjustment`
- `finalRecommendation`

まずは既存ロジックから近似で埋める。

### Phase 2: プリフロップGTOレンジ表

- 6max / 8max / 9max
- 50BB / 100BB / 20BB / 14BB
- RFI
- vs open call
- 3BET
- 4BET対応
- BB defend

### Phase 3: ポストフロップ代表ボード辞書

- A-high dry
- K-high dry
- paired board
- monotone
- two-tone connected
- low connected
- broadway connected
- 4-flush river
- paired river

### Phase 4: レンジ更新

- プリフロップアクション履歴から各プレイヤーの初期レンジを推定
- flop/turn/river の check/bet/call/raise/fold でレンジを削る
- リバー判断で相手のバリュー密度とブラフ候補を出す

### Phase 5: レビュー文を二段構造にする

```text
GTO基準:
実戦補正:
今回の推奨:
```

## 最終目標

Fish Tank Poker はリアルタイム完全ソルバーではなく、次を目指す。

```text
ソルバー由来の戦略辞書
+ レンジ更新
+ ライブ$2/$5補正
+ 初心者に伝わるレビュー文
```

完璧なGTOに近い理論土台を持ち、その上で実戦にズラす。
これにより、レビューは「なんとなくの経験則」ではなく、「GTOからどの方向にズラしたか」を説明できるようになる。
