# Fish Tank Poker 完成ロードマップ（実装設計図）

この文書は、Fish Tank Poker を「海外カジノ $2/$5 ライブキャッシュで勝てる下地を作るトレーニングソフト」として完成させるための実装設計図である。
実装エージェント（Codex 等）がこの文書だけで作業を進められることを目標にする。
理念は `AGENTS.md`、理論設計は `GTO_THEORY_BLUEPRINT.md`、実装履歴は `CHANGES.md` を正とする。

---

## 0. 現状の到達点と不足の全体像

到達済みのもの。

- リングゲーム / フロップトレーニング / トーナメント局面別の3モード。
- GTO / Live($2/$5) の二軸評価と自然文レビュー（`composeNaturalReview`, `coachReviewText`）。
- セッション統計（VPIP/PFR/3bet/CBet/WTSD 等、`updateSessionStats`）。
- Node テストハーネス `fish_tank_tests.js`（回帰124 + 理論23 + モード12 + プリフロップ6 + AI 2）。

不足しているもの（6領域）。

| 領域 | 不足 | 対応フェーズ |
|---|---|---|
| A. ライブ環境の再現 | ストラドル / レーキ / リンプ過多 / 9max検証 / 250BB超 | Phase 2 |
| B. 相手リード訓練 | タイプ隠し推定 / サイジングテル / 相手ノート | Phase 3 |
| C. 学習ループ | 弱点追跡 / 重点出題 / プリフロップ即答 / 再プレイ / bb100 | Phase 4 |
| D. 理論土台 | BLUEPRINT Phase 2〜4 の残り / サイズのモード連動 | Phase 5 |
| E. メンタル・メタ | ティルト / テーブル選択 / バンクロール教材 | Phase 6 |
| F. 保守性 | 1.13MB 単一HTMLの編集事故リスク / テストフレーク | Phase 0〜1 |

推奨実行順は Phase 0 → 1 → 2 → 3 → 4 → 5 → 6。
Phase 2〜4 が目的地への効果が最も大きい。
Phase 5 は評価精度の底上げであり、2〜4 と並行可能。

## 全フェーズ共通の作業ルール

1. 作業前に `node fish_tank_tests.js` で全グリーンを確認し、作業後も全グリーンを維持する。
2. 機能を1つ追加するたびに、再発防止テストを `runFishTankRegressionTests()` か `fish_tank_tests.js` の独立検証に追加する。
3. `CHANGES.md` に「何を・なぜ・どう直したか」を追記する。
4. `fish_tank_poker.html` の末尾 `</script></body></html>` の破損事故が既知のため、編集後に末尾タグの存在を確認する（Phase 0 でテスト化する）。
5. 評価ロジックの変更は GTO レイヤーとライブ補正レイヤーを混ぜない（`GTO_THEORY_BLUEPRINT.md` の基本原則）。
6. レビュー文は「なぜそのラインなのか」を口語で説明し、内部スコアを露出しない。

## Phase 0: 監査と足場固め（機能追加なし）

目的: 未確認事項を確定させ、以後の開発を安全にする。

### 0-1. HTML末尾ガードのテスト化
実装済み 2026-07-02: `fish_tank_tests.js` にHTML末尾ガードを追加。
`fish_tank_tests.js` に、読み込んだ HTML が `</script>`・`</body>`・`</html>` で正しく閉じていることを検証するテストを追加する。
受け入れ基準: 末尾を意図的に欠損させたコピーでテストが fail する。

### 0-2. テストフレークの解消
実装済み 2026-07-02: テスト用乱数とFAST用エクイティをseed固定化。
トーナメント reshove 監査に残る、`estimateEquity` のモンテカルロ乱数由来の非決定的 fail を解消する。
方針: テスト実行時のみ乱数をシード固定する（既存の `withSeededRandomForAudit` を流用）か、当該テストの試行回数を増やして分散を下げる。
受け入れ基準: `node fish_tank_tests.js` を10回連続実行して全て全グリーン。

### 0-3. 9max（フルリング）評価の監査
実装済み 2026-07-02: UTG/UTG+1/MPの代表ハンド監査を追加。
海外ライブは9maxが標準だが、開発は6max中心で進んできた。
`preflopPositionBucket` と `PREFLOP_RANGE_CHARTS` が9人時に UTG/UTG+1/MP を正しく厳しく扱うかを、代表ハンド（例: 9max UTG の KTo, ATo, 55, QJs）で監査する。
ズレがあればチャート側を修正し、9max ケースの回帰テストを追加する。
受け入れ基準: 9max の位置別オープン判定テストが追加され、全グリーン。

### 0-4. AIサイジング実装の監査
実装済み 2026-07-02: `aiDecide`/`aiSizedAction` を監査し、結果を `CHANGES.md` に記録。
`aiDecide` と `aiSizedAction` を読み、AI のベットサイズが手の強さと相関しているか（=サイジングテルが既に存在するか）を確定させる。
結果を `CHANGES.md` に記録する。
Phase 3-2（サイジングテル）の設計はこの監査結果を前提にする。

## Phase 1: 保守性基盤（推奨・省略可）

目的: 1.13MB 単一 HTML の編集事故を構造的に無くす。

### 1-1. ソース分割ビルド
実装済み 2026-07-03: `build.js` と `src/index.html` / `src/styles.css` / `src/app.js` を追加。
現時点では意味のある機能分割ではなく、既存の単一HTMLを安全に再生成するための第一段階として、HTML骨格・CSS・JS本体を機械的に分離した。
`node build.js --check` で、`src/` から再生成される `fish_tank_poker.html` が現在の配布HTMLと完全一致することを確認する。

`src/` に CSS / 各 JS モジュール / HTML骨格を分割し、Node スクリプト `build.js` の連結で従来と同一の単一 `fish_tank_poker.html` を生成する。
配布形態（単一HTMLをブラウザで開く）は変えない。
分割単位の目安: 評価エンジン / AI / トーナメントプロファイル / ライブキャッシュプロファイル / レビュー文生成 / UI / 回帰テストデータ。
受け入れ基準: ビルド生成物で `node fish_tank_tests.js` が全グリーン、かつ生成物のプレイに退行がない。
注意: このフェーズは差分が巨大になるため、単独コミットで行い、他の機能変更を混ぜない。
省略する場合、0-1 の末尾ガードを事故対策の代替とする。

## Phase 2: ライブ環境の再現

目的: 練習環境を海外 $2/$5 の実態に近づける。

### 2-1. ストラドル対応
仕様: 設定画面に「ストラドル」セレクタを追加する（なし / UTGストラドル 2xBB / 頻度ランダム）。
ストラドル時はプリフロップのアクション順がストラドラーの左から始まり、実質ブラインドが $2/$5/$10 になる。
評価側は実効スタック（BB換算が半減する）とオープンサイズ基準をストラドル基準で再計算する。
AI もストラドルポットでレンジをやや広げる（実効スタックが浅くなるため）。
実装ポイント: ゲームループのブラインド設定、`preflopSizePlan`、`preflopChartLookup` に渡す実効BB、レビュー文のBB表記。
受け入れ基準: ストラドルポットで (1)アクション順が正しい (2)評価が $10 基準のオープンサイズを推奨する、の回帰テスト追加。

### 2-2. レーキの組み込み
仕様: リングゲームにレーキ（例: 5% / キャップ $10、設定でOFF可）を実装し、ポット精算時に控除する。
評価側は小ポットの限界的な参加（リンプ・小さいコール）にレーキ分の減点を加える。
表示: ハンド履歴とレビューにレーキ額を表示し、「レーキがある場では小ポットの薄い参加が割に合わない」ことを説明できるようにする。
実装ポイント: ポット精算処理、`evalCall` / `evalBet` のプリフロップ限界コール分岐、`composeNaturalReview` のプリフロップ分岐。
受け入れ基準: レーキ有無で限界リンプの評価が変わる回帰テスト追加。

### 2-3. リンプ過多フィールド
仕様: AI難易度とは独立に「テーブルタイプ」設定を追加する（標準 / パッシブ・リンプ過多）。
リンプ過多テーブルでは、Live モードの AI が低〜中位ハンドで高頻度にリンプし、4〜5way ポットが常態化する。
評価側は既存のマルチウェイ補正（`mwFactor`、`liveCashMultiwayProfile`）がそのまま働くため、主な変更は AI 側で完結する。
アイソレーションレイズ（リンパーを挟んだレイズ）の推奨サイズを「3BB + リンパー1人につき1BB」の基準で `preflopSizePlan` に反映する。
受け入れ基準: リンプ過多設定で1ハンドあたり平均リンプ数が閾値を超えるシミュレーションテスト、およびアイソサイズの評価テスト追加。

### 2-4. ディープスタック
仕様: スタック選択肢に 300BB を追加する。
`liveCashSprProfile` と `deepSpr` 系プロファイルが 200BB 超で破綻しないかを監査し、必要なら「ディープではインプライドオッズとポジションの価値が上がり、ワンペアの価値が下がる」補正を強める。
受け入れ基準: 300BB の代表局面（セットマイン、ワンペア大ポット回避）の回帰テスト追加。

## Phase 3: 相手リード訓練モード

目的: ライブで最重要の「観察して相手を分類し、ズラす」技術を鍛える。exploit の核。

### 3-1. タイプ隠し + 推定 + 答え合わせ
仕様: 設定に「相手リード訓練」トグルを追加する。
ON のとき、ロスター（`renderRoster`）と `liveCashOpponentTypeProfile` 由来のタイプ表示（「コール多め」等）をレビュー文からも隠す。
ユーザーは各AIに対して「コール多め / ブラフ不足 / ブラフ多め / 標準」の推定ラベルをロスター上でいつでも設定できる。
一定ハンド数（例: 20ハンド）ごと、またはユーザーの要求時に答え合わせを表示し、正解率と「その相手のどの行動が根拠になり得たか」（ショーダウンしたハンド、ベットサイズ、フォールド頻度）を列挙する。
実装ポイント: `renderRoster` の表示分岐、推定ラベルの `localStorage` 保存、答え合わせ用に各AIの行動ログ（ショーダウン結果・サイズ・fold回数）をセッション内で集計する。
受け入れ基準: トグルONでタイプ文字列がUIとレビュー文のどこにも出ないことのテスト、答え合わせの正誤判定テスト追加。

### 3-2. サイジングテル
前提: Phase 0-4 の監査結果に従う。
仕様: Live モードの AI に「サイズテル持ち」の個体を混ぜる（例: 強い時だけ大きく打つ、ブラフ時に丸い数字を使う、など2〜3種）。
テルの有無と種類は相手リード訓練の答え合わせに含める。
GTO モードの AI にはテルを入れない（均衡挙動を守る）。
実装ポイント: AIプロファイル生成にテル属性を追加し、`aiSizedAction` のサイズ選択に反映する。
受け入れ基準: テル持ちAIのベットサイズ分布が手の強さで統計的に分かれるシミュレーションテスト追加。

### 3-3. 相手ノートUI
仕様: サイドパネルに「ノート」タブを追加し、AIごとに自由記述メモを保存できる（`localStorage`）。
ノートは同じ表示名のAIが再登場した際に引き継ぐ。
受け入れ基準: 保存・再表示の動作テスト（DOMはテストハーネスの偽DOMで検証可能な範囲でよい）。

## Phase 4: 学習ループの完成

目的: ミスを記録で終わらせず、定着と矯正につなげる。

### 4-1. 弱点の長期追跡
仕様: `primaryLessonForEval` が返す主テーマのカテゴリと quality を、ハンドごとに `localStorage` の統計へ蓄積する（既存 `sessionStats` と同じ永続化経路）。
「傾向」タブに弱点ランキング（カテゴリ別のミス率・件数・直近の推移）を表示する。
実装ポイント: `updateSessionStats` にカテゴリ集計を追加、`renderTrends` に表示を追加。
受け入れ基準: 既知のミスを含むハンドを流すとカテゴリ集計が加算される回帰テスト追加。

### 4-2. 弱点の重点出題（アダプティブドリル）
仕様: フロップトレーニングのシナリオ抽選（`_pickScenarioCat`）に「弱点優先」オプションを追加する。
4-1 の集計でミス率が高いカテゴリの重みを引き上げる（例: 上位2カテゴリの重みを2倍）。
プリフロップ起点の弱点（例: 3bet への対応）はフロップシナリオで表現できないため、対象はボードテクスチャ系カテゴリに限定し、限定した事実をUI上に明記する。
受け入れ基準: 集計を偽装した状態で抽選分布が偏ることのシミュレーションテスト追加。

### 4-3. プリフロップ即答ドリル
仕様: 新モード「プリフロップドリル」を追加する。
位置・人数・先行アクション（オープン / vs オープン / vs 3bet / スクイーズ）とハンドを提示し、Fold / Call / Raise を即答させ、`preflopChartLookup` と既存の評価ロジックで正誤と混合頻度を即時表示する。
1セッション20問、終了時に正答率とミス傾向を表示し、4-1 の弱点統計へ流し込む。
実装ポイント: 既存のテーブル描画を使わない軽量UIでよい（カード表示は `cardHTML` を流用）。
受け入れ基準: 出題→判定→集計の一連の回帰テスト追加。

### 4-4. ミス局面の再プレイ
仕様: ハンドレビューに「この局面をやり直す」ボタンを追加する。
同一のホールカード・ボード・スタック・アクション履歴を、ミスした判断の直前まで再現し、そこから再度プレイさせる（以後のAIの手札とランアウトは同一で固定する）。
実装ポイント: `hr`（ハンド履歴）は再現に必要な情報を既に保持している。回帰テストの `regressionHand` 生成系を再利用してゲーム状態を復元する。
受け入れ基準: 再プレイ局面の初期状態が元ハンドと一致することのテスト追加。

### 4-5. 収支と判断品質の分離表示
仕様: 「傾向」タブに bb/100（実収支）と判断スコア平均（品質）を並べて表示し、乖離しているときに「収支は分散、スコアが実力」という注記を出す。
用語 bb/100 は初出時に定義する（100ハンドあたりの獲得BB数）。
実装ポイント: `updateSessionStats` にハンドごとの収支(BB)を追加集計、`renderTrends` に表示追加。
受け入れ基準: 勝ち負けを固定したハンド列で bb/100 が正しく計算されるテスト追加。

## Phase 5: 理論土台の完成（BLUEPRINT残件 + 既知課題）

目的: `GTO_THEORY_BLUEPRINT.md` の Phase 2〜4 を完成させ、評価精度を底上げする。

### 5-1. プリフロップレンジ表の拡張（BLUEPRINT Phase 2 残り）
現状の `PREFLOP_RANGE_CHARTS` は open / flat / 3bet のみ。
追加するもの: vs 3bet（4bet / call / fold）、vs 4bet、スタック深度別チャート（100BB を正、50BB / 20BB / 14BB の差分を定義）、9max の位置バケット明示。
受け入れ基準: 各チャートの代表ハンド判定テスト（境界ハンドを含む）を追加。

### 5-2. 代表ボード辞書（BLUEPRINT Phase 3）
`boardTextureProfile`（分類器）の上に、9クラスの代表ボード辞書（A-high dry / K-high dry / paired / monotone / two-tone connected / low connected / broadway / 4-flush river / paired river）を定義する。
各クラスに、役割別（PFR IP / PFR OOP / コーラー）の基準ベット頻度とサイズ候補を持たせ、`solverBetProb` と `boardTextureSizePlan` の根拠として参照する。
受け入れ基準: 各クラス×役割の頻度出力テスト追加。

### 5-3. レンジ更新の完成（BLUEPRINT Phase 4）
`rangeActionUpdateProfile` を拡張し、ストリートごとの check / bet / call / raise / fold で相手レンジの「バリュー密度・ブラフ候補数」を更新して保持する。
リバー判断では更新済みレンジから必要勝率と比較できる推定EQを出し、レビュー文の「相手のレンジはバリューに偏ります」等の記述を定量根拠付きにする。
実ハンドを評価用レンジに混ぜない原則を守る。
受け入れ基準: 3バレルライン後のバリュー密度が1バレルより高いことの単調性テスト追加。

### 5-4. ポストフロップ・ベットサイズ推奨のモード連動（既知課題）
`suggestBet` / `boardTextureSizePlan` の推奨サイズを GTO / Live で分岐する。
方向: GTO は小さめ多頻度を許容、Live はバリュー時に大きめ・ブラフ頻度控えめ。
受け入れ基準: 同一局面でモードによりサイズ推奨が変わるテスト追加。

### 5-5. AIオープン幅のモード連動(既知課題)
現状はサイズのみ連動。Live はオープン幅をやや広く（特に BTN/CO）、GTO は基準どおりに分岐する。
受け入れ基準: AIプリフロップ検証にオープン幅の分岐テストを追加。

## Phase 6: メンタル・メタゲーム教材

目的: エンジン外の勝敗要因（ゲーム選択と自己管理）を教材化する。

### 6-1. 教材タブ
用語集と同じ形式で「ライブ実戦」タブを追加し、次のトピックを収録する。
テーブル/シート選択の基準、ストラドルポットの考え方、ティルトの兆候と離席基準、セッション終了判断、バンクロール基準（$2/$5 なら最低バイイン数の目安）、チップハンドリングとエチケット。

ゲーム開始時と終了時に任意表示のチェックリスト（体調・時間上限・ストップロス）を出せるようにする（設定でOFF可）。
受け入れ基準: 表示分岐の動作テスト。
