## 1. バグ修正（理論の誤り）

### ① ハンドランク表（HAND_RANK_169）の並び間違い
169手のエクイティ順テーブルに2か所の破綻があり、`handFrac`（=全評価の土台）を歪めていた。

- `QTs` がランク63（上位27%）で **`QTo`（上位15.5%）より弱い**扱い → スーテッドがオフスーツに負ける矛盾。`KTs`並みの上位24に修正。
- `94s` がランク166（ほぼ最弱）→ `95s`近辺（76）へ修正。

影響: COオープンの `QTs` を「ルース」と誤って減点し `QTo` を許容する等の逆転指導を解消。

### ② BB防御MDFの反転
`defendPct=(1-eMDF)` となっており、**小ベットほど降り・大ベットほど受ける**というMDFと真逆の挙動。`defendPct=eMDF` に修正し、ベットが小さいほど広く守る正しい方向へ。

### ③ リバーの実現率（realization）二重割引
`evalCall`/`evalFold`/`postflopContext` でリバーにも実現率係数（OOP×0.82、マルチウェイ×0.62 等）を掛けていた。リバーは残りカードが無く実現率の概念が存在しない（EQ=ショーダウン勝率）。レンジ補正は別系統（pressure）で行うため二重ペナルティだった。リバー時は実現率を1.0に固定。

### ④ estimateEquity の精度・決定論化
- 相手にランダム2枚を配るモンテカルロ。完成リバー×単独相手は **全相手2枚を厳密列挙**して決定論化（同一局面で評価がブレない）。
- タイをそれまで「勝ち」に数えていたのを **0.5** に修正。
- 評価呼び出しの試行回数を 400→2000 に増やしノイズを低減（AIの250は速度のため据え置き）。

### ⑤ handRole のクラッシュ（ハーネスで発見）
ツーペア分岐で変数 `madeDraw` が未定義のまま参照され、**ポケットペア×ペアボード（例: AA on 8-8-5、55 on 8-8-3）で必ず `ReferenceError` を投げ評価全体がクラッシュ**していた。定義を追加して修正。

## 2. GTO / Live レンジ判定トグル

設定画面に「レンジ判定」セレクタを追加（既定=ライブ$2/$5、選択でGTO）。`localStorage` で永続化。`RANGE_MODE` グローバル + `getRangeMode()/setRangeMode()`。

設計原則: **Live=$2/$5基準（既存コードは既にライブ寄りにチューニング済→現状維持で回帰を守る）、GTO側と"場の人数"で差を付ける。**

トグルが貫通する全レイヤー:

| レイヤー | Live | GTO |
|---|---|---|
| **プリフロップ3bet幅** | 基準 | 広い（ポラーなブラフ3bet） |
| **非IPフラット** | 基準 | 圧縮（3bet-or-fold寄り） |
| **スクイーズ**（間にコール有） | バリュー寄せで締め | 同左 |
| **マルチウェイ・フラット** | 投機系広げ/ドミネート系締め | — |
| **プリフロップ・サイズ** | 大きめ（CO 3BB / 3bet 3.4〜4倍） | 小さめ（CO 2.2BB / 3bet 3〜3.5倍） |
| **実効EQ（プレッシャー割引）** | 強い割引 | 割引を約60%に緩和 |
| **品質バッジ** | bad に倒す | EVを尊重（+EVはok、-EVのみbad） |
| **レビュー本文の結論** | フォールド寄り | コール寄り |
| **AI相手の3bet頻度** | ブラフ3bet ×0.65 | ×1.15 |
| **AI相手のコールド/フラット** | ×1.20（受け身） | ×0.78 |
| **AI相手のリンプ** | 後ろ位置で稀に混在 | 無し |
| **AI相手のオープンサイズ** | 大きめ（2.7〜3.3BB） | 小さめ（2.2〜2.5BB） |
| **AI相手のポストフロップ・ブラフ** | リバー×0.45 / 他×0.65 | 均衡 |

### 整合性の核
同一のリバー大ベット局面（TPTKが3バレル受け、OOP）で:

| | 実効EQ | onePair判定 | 品質 | 本文の結論 |
|---|---|---|---|---|
| Live | 30% | bad | bad | 6:4〜7:3でフォールド寄り |
| GTO | 47% | border | ok | 8:2でコール寄り |

相手AIの挙動・評価ロジック・実効EQ・品質・説明文が全層で一致。

## 3. ハンドレビュー文の自然化

従来は「フラグごとの断片」を連結し、二重句点・角括弧タグの羅列・重複でギクシャクしていた。

全weight適用後に走る `composeNaturalReview` を新設し、リングのポストフロップ コール/フォールド/ベットを **「場面 → 手 → 相手の傾向 → 数字 → 結論（混合比）→ 助言」** の自然な一段落に再構築。詳細な監査内容は本文から外しメタチップ側へ集約（情報は失わない）。混合は「ほぼ互角(5:5)」「6:4でフォールド寄り」のように **数値比+言葉** で表示。

**Before**: `コール（EQ約72%、必要30%）…【$2/$5実戦レンジ補正】…。。…【リング文脈】…【ワンペア監査】…【リバー金額判断】…`（断片の山）

**After**: `リバー。トップペア強キッカー(エース)で、相手の大きめのベット(ポットの74%)をアウトオブポジションで受ける場面です。相手は3ストリート続けて打っており、レンジはバリューに偏ります。…実効エクイティは約30%。コールに必要な勝率は30%です。GTO上はほぼ互角(5:5前後)ですが、ライブ$2/$5では…6:4〜7:3でフォールド寄りで、相手が明らかにブラフ過多のタイプの時だけコールに回しましょう。`

### 表示経路の修正（重要）
当初は自然文を `ev.comment` に書いていたが、画面の**主役レビューは別レンダラ `coachReviewText(ev)`**（`ev.comment` は折りたたみ「詳細データ」内の小さい灰色表示）だった。そのため自然化が主役に反映されていなかった。`composeNaturalReview` の出力を `ev.naturalReview` にも保存し、`coachReviewText` がそれを最優先で返すように修正。これで主役レビューが自然文になった。

### プリフロップも自然化
当初ポストフロップのみ対象だったが、`composeNaturalReview` にプリフロップ分岐を追加（`liveCashSpotProfile` の label/policy/risk から「場面→手→理由→補足」を構成）。冗長な二重結論を排除。

### 文法バグ修正
`recommendationClose` で、推奨文が「〜ください」で終わる完結文の時に「を意識してください」を二重付加して「使い分けてください**を意識してください**」のように壊れていたのを修正。

### アンダーペア／ボードペアの説明取り違え（ユーザー指摘）
`handRole` は44等のポケットアンダーペアを正しく `under_pair` と分類していたが、下流のコメント生成が `weakPair`（board_pair / under_pair / bottom_pair 等）を一括で「ボードのペアにキッカーが乗るだけ」と説明していた。アンダーペア（自分のポケットペア）には誤り。`pairTier` で分岐し、`board_pair` のみ「ボードペア・キッカー」、それ以外は「ボードの上位カードに負けている弱いワンペア」と説明するよう修正。再発防止テストを追加。

### チェック・ラインも自然化
`composeNaturalReview` をチェックにも対応。「フロップ。中・低ペアでインポジション、チェックの場面です。生EQ…実効EQ…。ショーダウン価値を守るチェックが自然です。」のように、旧来の「チェックは適切な選択です。チェックを許容。…」という冗長連結を解消。

### 小ペアのオープン範囲（ユーザー指摘）
ポケットペアがオープン上限（MP 上位約15%）から外れ、44 MP オープン等が過剰減点されていた。セットマイン価値を踏まえ、オープン／オープンフォールド判定にペアのフロアを追加（Live=全ペア開く、GTO=22 UTG 等のみ境界）。

### 主テーマが局面に合わない（ユーザー指摘 #5）
主テーマ（primaryLesson）が、(1) 範囲内の標準的なプリフロップ・オープンを「整理すべき主題」として乗っ取る、(2) IP・ヘッズアップの局面で「OOP・マルチウェイ・動的ボード」という汎用OOP文を流用する、という2つのズレがあった。修正: (1) 範囲内の標準オープン/レイズ（quality が bad でなく減点 < 8）は主テーマ候補から除外、(2) 主導権テーマを `isIP`／ヘッズアップに応じて文面を切り替え（IP では「インポジションの主導権をどう使うか／スケアカードでの薄バリューとポット管理」）。結果、例の #6 は主テーマが「リバー判断を相手のラインから決める」（4枚スペードでトリップスを降りる規律＝このハンドの本当の学習点）に正しく移った。再発防止テストを追加。

### 脆弱ハンドのスケアボード・チェックバック過剰減点（ユーザー指摘）
`evalCheck` が rawEQ の高さ＋レンジ優位だけで「強い手のチェック＝EV損失」と判定し、トリップス弱キッカー（例: J9 on K-J-J-8s）の動的ターンでのチェックバックを -25点/bad と過剰に罰していた。`handRole` は当該手を `role='value'`＋`isVuln=true` と正しく見ているのに採点へ反映されていなかった。`isVuln` の value/medium 手を、フラッシュ寄り（flushy≥3）／ペア／動的なフロップ・ターンでチェックバックする場合は正当なポット管理として `betProb` を下げ、減点を緩和（例の局面で -25→-5、bad→ok）。表示文も「ベット寄りだが脆弱な手のチェックバックも戦略内」に。再発防止テストを追加。
